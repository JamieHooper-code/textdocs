EMAIL CLEANUP FLOW — BUILD GUIDE
=================================
Power Automate (cloud) + Excel config on OneDrive
Date: April 2026


== OVERVIEW ==

Two separate flows. Keeping them separate means if one breaks it doesn't
affect the other, and each can be scheduled independently.

  Flow 1 — "Email Cleanup: Apply Rules"
    Runs on a schedule (e.g. every 6 hours).
    Reads the Rules sheet. For each enabled rule, finds matching emails
    older than MinAgeHours. Moves them to a folder or to the Staging folder.
    Writes a row to the Run Log sheet for each email touched.
    Sends a summary email if SendRunSummary = yes.

  Flow 2 — "Email Cleanup: Hard Delete Staged"
    Runs once daily.
    Reads the Staging folder. Hard-deletes anything older than
    HardDeleteAfterHours. Writes to the Run Log.

SAFETY FEATURES BUILT IN:
  - DryRunMode: flow logs everything but takes no action until you turn it off
  - NeverHardDelete: when on, nothing is ever permanently deleted — emails go
    to a "Deleted by Script" graveyard folder instead. On by default.
  - Staging folder: emails are never acted on directly — always staged first
  - HardDeleteAfterHours: staged emails sit for X hours before processing
  - GlobalMinAgeHours: no rule can fire on emails younger than this floor
  - MaxEmailsPerRule: caps how many emails can be processed per rule per run
  - Run Log sheet: permanent record of every action taken


== PREREQUISITES ==

Before building the flows:

  1. Open email_cleanup_config.xlsx. Review the rules and confirm the Settings
     sheet looks correct. Make sure DryRunMode is set to "yes" — leave it there
     until testing is done.

  2. In Outlook, create these two folders at the top level of your mailbox:
       - "Pending Delete"    (or whatever StagingFolderName is set to in Settings)
       - "Deleted by Script" (or whatever SafeDeleteFolderName is set to in Settings)
     To create a folder: right-click your mailbox name in the folder list > New Folder.

  3. Upload email_cleanup_config.xlsx to OneDrive.
     Suggested path: OneDrive > Documents > Email Cleanup > email_cleanup_config.xlsx
     Note the exact path — you will need it when building the flows.

  4. Make sure Power Automate is accessible:
     Go to make.powerautomate.com and sign in with the work Microsoft account.
     If the account doesn't have access, IT may need to enable it.


== FLOW 1: EMAIL CLEANUP — APPLY RULES ==

--- Create the flow ---

1. Go to make.powerautomate.com
2. Click "Create" in the left sidebar
3. Choose "Scheduled cloud flow"
4. Name it: "Email Cleanup: Apply Rules"
5. Set recurrence: every 6 hours (or daily — adjust to preference)
6. Click Create

--- Add steps ---

The steps below map to what Power Automate calls "actions." Add them in order
using the "+ New step" button after each one.

STEP 1 — Read Settings
  Action: "List rows present in a table" (Excel Online - Business connector)
  Location: OneDrive for Business
  Document Library: OneDrive
  File: [browse to email_cleanup_config.xlsx]
  Table: SettingsTable

  After this step, add a "Compose" action to build a settings lookup object.
  This makes it easier to reference settings by name later.
  Action: Compose
  Inputs (paste this expression):
    union(
      json('{}'),
      reduce(
        outputs('List_rows_present_in_a_table')?['body/value'],
        json('{}'),
        addProperty(current, item()?['Setting'], item()?['Value'])
      )
    )
  Name this step "Settings Object" using the "..." rename option.

  NOTE: If the expression above is too complex to paste, a simpler approach
  is to add individual "Get a row" actions for each setting you need,
  filtering by the Setting column. Slower but easier to read.

STEP 2 — Check Dry Run Mode
  Action: Condition
  Left side:  outputs('Settings_Object')?['DryRunMode']
  Operator:   is equal to
  Right side: yes

  This creates an IF/ELSE branch. All actual email actions go in the NO branch.
  The YES branch just logs "DRY RUN — no actions taken."

STEP 3 (inside NO branch) — Read Rules
  Action: "List rows present in a table"
  Same file as Step 1, Table: RulesTable

STEP 4 — Loop over rules
  Action: "Apply to each"
  Select output: value (from the List rows step above)

  Inside the loop, add these sub-steps:

  STEP 4a — Check if rule is enabled
    Action: Condition
    Left:  items('Apply_to_each')?['Enabled']
    Op:    is equal to
    Right: yes

    Everything below goes inside the YES branch of this condition.

  STEP 4b — Calculate cutoff time
    Action: Compose
    Inputs (expression):
      addHours(
        utcNow(),
        mul(-1, max(
          if(endsWith(items('Apply_to_each')?['MinAge'], 'h'),
            int(substring(items('Apply_to_each')?['MinAge'], 0, sub(length(items('Apply_to_each')?['MinAge']), 1))),
            if(endsWith(items('Apply_to_each')?['MinAge'], 'd'),
              mul(int(substring(items('Apply_to_each')?['MinAge'], 0, sub(length(items('Apply_to_each')?['MinAge']), 1))), 24),
              if(endsWith(items('Apply_to_each')?['MinAge'], 'm'),
                mul(int(substring(items('Apply_to_each')?['MinAge'], 0, sub(length(items('Apply_to_each')?['MinAge']), 1))), 720),
                int(items('Apply_to_each')?['MinAge'])
              )
            )
          ),
          int(outputs('Settings_Object')?['GlobalMinAgeHours'])
        ))
      )
    Name this: "Cutoff Time"
    This parses the MinAge field: h = hours, d = days (x24), m = months (x720).
    A plain number with no suffix is treated as hours for backwards compatibility.
    The max() ensures GlobalMinAgeHours is always respected as a safety floor.

  STEP 4c — Get matching emails
    Action: "Get emails (V3)" (Office 365 Outlook connector)
    Folder: [expression — uses SourceFolder if set, otherwise defaults to Inbox]
      if(or(empty(items('Apply_to_each')?['SourceFolder']), equals(items('Apply_to_each')?['SourceFolder'], 'none')),
         'Inbox',
         items('Apply_to_each')?['SourceFolder'])
    Fetch Only Unread Messages: No
    Include Attachments: No
    Top: [value of MaxEmailsPerRule from settings — use expression:
          int(outputs('Settings_Object')?['MaxEmailsPerRule'])]

    Filter Query (expression — build this carefully):
      concat(
        if(or(empty(items('Apply_to_each')?['FromAddress']), equals(items('Apply_to_each')?['FromAddress'], 'none')),
           '',
           concat("from/emailAddress/address eq '", items('Apply_to_each')?['FromAddress'], "' and ")),
        if(or(empty(items('Apply_to_each')?['SubjectContains']), equals(items('Apply_to_each')?['SubjectContains'], 'none')),
           '',
           concat("contains(subject,'", items('Apply_to_each')?['SubjectContains'], "') and ")),
        "receivedDateTime lt ", outputs('Cutoff_Time')
      )

    NOTE: This builds an OData filter string combining whichever fields are
    filled in. OData is just a standard query language for filtering data —
    you don't need to know it deeply, just paste the expression above.

    If the filter expression is causing errors, a simpler fallback:
    - Leave Filter Query blank (gets all recent emails)
    - Add condition steps inside the email loop to check sender/subject manually
    - Less efficient but easier to troubleshoot

  STEP 4d — Loop over matched emails
    Action: "Apply to each" (nested inside the rules loop)
    Select output: value from the Get emails step

    STEP 4d-i — Move email
      Action: "Move email (V2)"
      Message Id: [ID from current email item]
      Destination Mailbox Address: [the account's email address]
      Destination Folder Id: items('Apply_to_each_2')?['DestinationFolder']

      NOTE: Both "move" and "pending-delete" rules use DestinationFolder directly.
      For pending-delete rules, DestinationFolder contains the staging folder name
      (e.g. "Pending Delete") — Flow 2 will pick it up from there.
      No branching needed here.

    STEP 4d-ii — Write to Run Log
      Action: "Add a row into a table" (Excel Online - Business)
      File: same config file
      Table: RunLogTable
      Row values:
        Timestamp:         utcNow()
        RuleName:          items('Apply_to_each')?['RuleName']
        EmailSubject:      items('Apply_to_each_2')?['subject']
        FromAddress:       items('Apply_to_each_2')?['from']
        EmailAge_Hours:    div(sub(ticks(utcNow()), ticks(items('Apply_to_each_2')?['receivedDateTime'])), 36000000000)
        Action:            items('Apply_to_each')?['Action']
        DestinationFolder: items('Apply_to_each')?['DestinationFolder']
        DryRun:            no
        Notes:             [leave blank]

STEP 5 — Send summary email (optional)
  Add a condition: SendRunSummary equals yes
  YES branch:
    Action: "Send an email (V2)"
    To:      [SummaryEmailTo from settings]
    Subject: Email Cleanup Run Summary — [utcNow()]
    Body:    [Free text — include rule counts, total emails actioned, etc.
              Power Automate doesn't have great loop counters built in,
              so a simple "Flow ran successfully" message is fine to start]


== FLOW 2: EMAIL CLEANUP — HARD DELETE STAGED ==

1. Create a new Scheduled cloud flow
   Name: "Email Cleanup: Hard Delete Staged"
   Recurrence: Once daily (e.g. 3am)

STEP 1 — Read Settings
  Same as Flow 1 Step 1 — list rows from SettingsTable, build Settings Object.

STEP 2 — Check Dry Run Mode
  Same condition check as Flow 1.

STEP 3 (inside NO branch) — Get emails from staging folder
  Action: "Get emails (V3)"
  Folder: [StagingFolderName from settings — e.g. "Pending Delete"]
  Filter Query:
    concat("receivedDateTime lt ", addHours(utcNow(), mul(-1, int(outputs('Settings_Object')?['HardDeleteAfterHours']))))

STEP 4 — Check NeverHardDelete setting
  Action: Condition
  Left:  outputs('Settings_Object')?['NeverHardDelete']
  Op:    is equal to
  Right: yes

  YES branch (safe mode — move to graveyard folder instead of deleting):
    Action: "Apply to each" over email results
      STEP — Move email to safe delete folder
        Action: "Move email (V2)"
        Message Id: [ID of current email]
        Destination Folder Id: [SafeDeleteFolderName from settings]
      STEP — Write to Run Log
        Same as Flow 1 STEP 4d-ii, but Action = "safe-delete", DryRun = no

  NO branch (actual deletion — only if NeverHardDelete = no):
    Action: "Apply to each" over email results
      STEP — Delete email
        Action: "Delete email (V2)"
        Message Id: [ID of current email]
      STEP — Write to Run Log
        Same as Flow 1 STEP 4d-ii, but Action = "hard-delete", DryRun = no


== TESTING CHECKLIST ==

Before setting DryRunMode to "no":

  [ ] Flow 1 runs without errors in dry run mode
  [ ] Run Log sheet shows correct emails that WOULD have been moved
  [ ] Ages in the EmailAge_Hours column look right
  [ ] RuleNames in the log match what you expect
  [ ] No unexpected emails appearing (check for overly broad rules)
  [ ] Flow 2 runs without errors in dry run mode
  [ ] Staging folder ("Pending Delete") exists in Outlook
  [ ] Safe delete folder ("Deleted by Script") exists in Outlook

When enabling for real (recommended order):
  [ ] Keep NeverHardDelete = yes (this should stay on indefinitely for most users)
  [ ] Change DryRunMode to "no" in Settings sheet
  [ ] Run Flow 1 manually once and watch it
  [ ] Check the staging folder in Outlook — correct emails moved there?
  [ ] Wait for Flow 2 to run (or trigger manually)
  [ ] Check "Deleted by Script" folder — emails should be accumulating there
  [ ] Check Run Log after each run for the first week
  [ ] Only set NeverHardDelete = no if permanent deletion is explicitly wanted
      and you have verified the flow is behaving correctly for several weeks


== HANDING OFF TO THE CLIENT ==

Files to send:
  1. email_cleanup_config.xlsx  (generated by generate_config.py)
  2. This guide

They need to do:
  1. Upload the Excel file to their OneDrive
  2. Create the "Pending Delete" folder in Outlook
  3. Import the flows (once you export them from your test environment)
     OR follow this guide to build them directly in their account
  4. Authorize the Outlook and Excel connections when prompted (one-time sign-in)
  5. Run in dry run mode, review the log, then enable

After setup, the only file they ever need to touch is the Excel config.
No Power Automate UI interaction required for day-to-day changes.


== EXPORTING AND IMPORTING FLOWS ==

To export a flow (from your account, to send to someone):
  1. Open the flow in Power Automate
  2. Click the "..." menu at the top right
  3. Export > Package (.zip)
  4. Give it a name and click Export
  5. Send the .zip file

To import a flow (recipient's side):
  1. Go to make.powerautomate.com
  2. My flows > Import > Import Package (Legacy)
  3. Upload the .zip
  4. For each connection shown, click "Select during import" and sign in
  5. Click Import
  6. Open the imported flow and update the Excel file path to match their OneDrive

NOTE: Connection references (the sign-in to Outlook, Excel, etc.) cannot be
pre-configured — the recipient must authorize them with their own account.
This is a one-time step that takes about 2 minutes.


== REGENERATING THE CONFIG FILE ==

You should not need to do this. The Excel file is the only thing you need to
touch for day-to-day changes — just edit it directly.

If the file is ever lost or corrupted and needs to be rebuilt from scratch:

  1. Install Python 3 if not already installed (python.org)
  2. Run: pip install openpyxl
  3. Run: python generate_config.py
  4. This produces a fresh email_cleanup_config.xlsx with default example rules.
     Re-enter your actual rules before uploading to OneDrive.
