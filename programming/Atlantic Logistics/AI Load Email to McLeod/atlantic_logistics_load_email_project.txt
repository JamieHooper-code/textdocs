CLAUDE'S ANALYSIS (2026-04-02)
--------------------------------------------------------------------------------

WHAT THEY'RE ASKING FOR:
  - A dedicated email inbox where customer load emails get forwarded
  - LLM parses each email → extracts structured load data
  - Posts loads to McLeod TMS (Transportation Management System) via API
  - Creates "Subject Orders" in McLeod
  - Keeps orders current as new emails arrive:
      * New single load → add it
      * Replacement load list → delete old ones, post new ones
  - Customer profiles: pre-fill defaults per sender (e.g., trailer type)
  - Future phase: pricing API integration to check margin before posting

WHY IT'S A GOOD OPPORTUNITY:
  - Clear, measurable ROI — freight brokers post dozens of load lists daily
  - The Optimus Steel example is a structured HTML table — easy for LLMs to parse
  - Well-scoped: email → parse → API post is a standard pipeline
  - Phase 2 (pricing API) is a natural extension already mentioned by client

TECHNICAL STACK (recommended):
  - Email monitoring: Gmail API or dedicated forwarding inbox (e.g., SendGrid inbound parse)
  - LLM parsing: Claude API (claude-sonnet-4-6 or haiku for cost)
  - TMS posting: McLeod REST API → Subject Orders endpoints
  - State tracking: simple database (SQLite or Postgres) to track what's been posted per customer
  - Customer profiles: JSON or DB table mapping sender email → defaults (trailer type, broker ID, etc.)

KEY TECHNICAL CHALLENGES:
  1. IDEMPOTENCY / LIST DIFFING — hardest problem
     - Must determine: "is this a full replacement list or an additive single load?"
     - Approach: LLM classifies email intent + compare against current posted loads for that sender

  2. EMAIL FORMAT VARIABILITY
     - Optimus Steel has a clean HTML table (easy)
     - Other customers may send: messy Word docs, PDFs, image attachments, free-form text
     - Need robust extraction that handles multiple formats

  3. McLEOD API ACCESS
     - Need API docs + sandbox credentials from Atlantic Logistics before scoping
     - McLeod's API (PowerBroker/LoadMaster) can vary by version/config
     - This is the critical dependency — confirm before committing to timeline

  4. ERROR HANDLING + HUMAN REVIEW
     - A misread rate or wrong origin/destination could cause real business problems
     - Consider: low-confidence extractions → flag for human review before posting
     - Confidence threshold + review queue recommended

QUESTIONS TO ASK ROBERT BEFORE STARTING:
  1. Do you have McLeod API access + docs + sandbox environment?
  2. What version of McLeod are you running (PowerBroker? LoadMaster?)
  3. How many customer distros are we talking about initially? 5? 50?
  4. Are all customer emails as clean/structured as Optimus Steel, or is there variety?
  5. Who posts loads in McLeod manually today? (to understand the "post as" user/auth)
  6. Is human review acceptable, or must this be fully automated?
  7. What's the expected email volume per day?
  8. Pricing API — when is that available, and do we build for it now or add later?

SUGGESTED APPROACH / PHASES:
  Phase 1 — MVP:
    - Single customer (Optimus Steel as pilot)
    - Gmail API polling every 5-10 min
    - Claude extracts load data from email
    - Manual review step before McLeod post
    - McLeod API: create Subject Orders

  Phase 2 — Full Automation:
    - Multi-customer support with profiles
    - Automatic list diffing (new vs. replacement)
    - Auto-post high-confidence results, flag low-confidence
    - Handling PDF/image attachments (vision API)

  Phase 3 — Pricing Integration:
    - Call pricing API before posting
    - Only post if margin threshold met
    - Reporting dashboard
