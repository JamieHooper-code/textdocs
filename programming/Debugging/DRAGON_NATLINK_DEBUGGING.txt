WHY CASTER COMMANDS WORK BUT DICTATION DOESN'T
-----------------------------------------------
Caster grammar commands:
  speech -> Dragon recognizes -> fires Python gotResults() callback via Natlink -> AHK/key actions
  No UI Automation needed. Purely callback-based.

Dragon native dictation:
  speech -> Dragon recognizes -> NatText component tries to inject text into focused window
  Requires dgnuiasvr.exe (UI Automation Server) to find the target text field and insert text.
  If dgnuiasvr.exe is dead or broken, dictation silently fails or hangs.

TEXT INJECTION METHOD BY APP TYPE
----------------------------------
Not all apps use the same injection path. This explains why Notepad works when Chrome doesn't:

  Win32 EDIT controls (classic Notepad, legacy apps):
    Dragon injects text via WM_SETTEXT / WM_CHAR messages directly to the window.
    Does NOT use dgnuiasvr.exe. Works even when UIA servers are broken.

  Microsoft Word:
    Dragon uses its own COM API. Does not use dgnuiasvr HOOKDLL.

  Modern apps (Chrome, VS Code, Electron, Explorer, Notepad++):
    Dragon uses dgnuiasvr.exe + HOOKDLL + SendInput thread to inject keystrokes.
    If dgnuiasvr.exe's SendInput thread is dead, ALL of these fail silently.

CONFIRMED: Notepad (classic Win32) works after crash/restart. Chrome fails.
This is NOT an orphan DLL problem — it is a dead dgnuiasvr.exe SendInput thread.

CONFIRMED ROOT CAUSE
--------------------
dgnuiasvr.exe (32-bit, PID 7396) had a dead SendInput thread.

The thread failed to properly initialize when dgnuiasvr auto-restarted at 19:07:54.
The 0x800705a4 error appeared at 19:08:24 — only 30 seconds after dgnuiasvr started —
meaning any natspeak connecting to that instance (whether before or after PID 9880) would
be connecting to a broken UIA server. Dragon recognized speech perfectly at 19:15:10.711
but text injection failed 8ms later with the same error.

WHY ONLY REBOOT FIXED IT
-------------------------
Before this session's investigation, RebootDragon() killed dgnuiasvr_x64.exe but NOT
dgnuiasvr.exe (32-bit). The 32-bit server survived every soft restart, holding its dead
SendInput thread state. When natspeak relaunched, it found the existing dgnuiasvr.exe and
reused it — inheriting the broken state. A full system reboot killed everything, forcing
a clean dgnuiasvr initialization.

DISPROVED THEORIES
------------------
1. ROT/COM stale registration: DISPROVED.
   Waited 5+ minutes, killed all processes including dgnsvc.exe — still broken.

2. HOOKDLL orphaned in target processes: PARTIALLY DISPROVED.
   A fresh Chrome instance (never had Dragon's DLL injected) also fails.
   Notepad works (different injection path — WM_ messages, not UIA).
   The failure is in dgnuiasvr.exe itself, not in target app processes.

3. DragonSvc service holding stale state: DISPROVED.
   Stop/start DragonSvc had no effect. dgnsvc.exe is not involved in UIA server state.

NOTE ON dgnsvc.exe:
  dgnsvc.exe is a Windows Service — when killed with taskkill, the Service Control Manager
  automatically restarts it within seconds. It cannot be "cleared" by killing it.

THE FIX
-------
The fix is to kill BOTH dgnuiasvr.exe (32-bit) and dgnuiasvr_x64.exe before relaunching
natspeak. When natspeak starts with no existing dgnuiasvr.exe, it spawns a fresh one that
properly initializes its SendInput thread.

KNOWN ERROR CODES IN LOGS
--------------------------
0x800705a4 = ERROR_INVALID_THREAD_ID (Win32 1444)
  -> PostThreadMessage failed because the target thread is dead
  -> Dragon's SendInput thread is gone (dgnuiasvr.exe crashed or not running)

0x80010108 = RPC_E_DISCONNECTED
  -> COM object has disconnected from its clients
  -> Dragon is calling a COM method on an object that's no longer alive (dead UIA server)

0x800706ba = RPC_S_SERVER_UNAVAILABLE
  -> RPC server unavailable — floods the log when dgnuiasvr.exe is killed while Dragon is running

0x80020003 = DISP_E_MEMBERNOTFOUND
  -> Dragon's hook DLL trying to call IAccessible on a Windows Shell control that doesn't have that method
  -> Seen in ShellExperienceHost Dragon.log — not a crash-causing error, just cosmetic
